## angles
- The leverage zone of the system sits in a fixed-form context (vendor contracts, two stacks, state legal) -> The shared header code nobody owns is the precise zone where small structural fixes propagate to all 60+ screens -> Assign ownership of shared header component as first act before screen-by-screen work, treating it as the highest-leverage point
- Rank evenness is audible only in passage, not in individual tests -> Users navigate task paths (housing application start to submit, food assistance start to submit) not screens, so failures compound cumulatively across forms and only path-level testing catches these seam failures -> Run full task paths with screen reader before finalizing remediation priority order rather than sorting 140 failures by individual screen severity
- Attack and steady-state are distinct failure modes -> Users on prepaid data and five-year-old phones fail at onset (form load on slow connection before layout settles, before screen reader fires) where meaningful landmarks, skip links, and sensible focus order matter, which differs from steady-state usability and doesn't require wording changes -> Test throttled connections separately, treating the onset phase as a distinct failure class outside standard audits and legal cycles
- Root-cause patterns precede voicing work -> The 140 failures likely cluster by source: shared template, shared component, vendor default, or wording problem appearing repeatedly across screens represents one defect, not many -> Produce a root-cause map organized by source not by screen; showing legal a single cross-screen wording fix instead of twenty individual ones compresses six-week cycles
