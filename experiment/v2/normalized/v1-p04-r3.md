## angles
- Screen and siting are separate contamination sources with different serviceability; parse cost per message likely conflates message complexity (environment) with deserializer implementation (instrument) -> when an instrument runs hot, isolating the source determines which one you can actually touch -> decompose per-message parse cost by implementation (alternative JSON parsers, lazy AST, byte-level field extraction) before attempting siting changes
- Consumers read every byte and build complete object graphs but scorers use only subsets of fields; this is like a sensor transducing the full thermal environment when only one property is relevant -> the relation is structural and you pay full cost for partial signal use -> deploy partial or lazy deserialization (parse only the fields the scorer consumes) to reduce material surface the sensor responds to without changing the environment
- Backlog outputs have temporally displaced accuracy like high-thermal-mass siting producing readings that reflect hours-old heat distribution; the lag is not uniform noise but has a predictable daily envelope -> since contamination phase is known, use it to triage routing -> during backlog windows, queue position tells you whether a score arrives inside or outside decision windows; route or discard on that basis rather than draining uniformly, and label stale readings for different handling than fresh ones
