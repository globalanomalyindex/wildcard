## angles
- Consumer-group lag is tracked in message count (20M messages), but SLA protection depends on elapsed time in backlog → measure estimated lag-seconds per partition (lag-count × p95 deserialization latency) instead → set load-shedding threshold and alerting at 45 seconds of backlog time, not message count; this costs an afternoon to wire and requires no infra changes
- 48 partitions apply identical full JSON deserialization regardless of current lag → switch to selective parsing when partition lag exceeds 30 seconds, extracting only the 3–5 fields the fraud model reads (amount, merchant ID, card hash, velocity bucket, country) via Jackson streaming or simdjson, and skip the rest → typical CPU savings of 40–70% on fraud-scoring payloads, tunable per partition
- All 48 partitions push every message through the full scoring pipeline regardless of prior-probability risk → add a pre-filter stage on first 50–80 bytes (enough to extract amount and merchant-category code) that routes low-signal messages to pre-computed default scoring and high-signal messages to full deserialization → enable only when backlog exceeds threshold, disable when drained; no format change, no new partitions
