## angles
- The engineering workflow already has structural joints (PR merges, deploy triggers, closed tickets) where documentation actions can attach themselves -> rather than asking for new policy authority, identify an existing signal that reliably marks a shipped change -> hook documentation updates to that signal, like a claque cues off the note-end event already in the score.
- Stale pages surfacing in search rankings at the wrong moment expose the documentation system as mechanically failed -> when content misfires into the customer experience (someone acting on outdated information), it becomes visible as a two-day lockout or support ticket -> consider whether the help center's search ranking offers a lever to suppress or mark content by "last reviewed" date, so staleness stops before verification can happen.
- Documentation triage should prioritize by failure cost, not by age or topic -> an auth guide where one stale section caused a two-day outage is load-bearing (aria-level); a 2023 Confluence space is likely full of recitative whose staleness is cheaper -> allocating limited attention to the documents that would produce the most expensive downstream event gives a defensible prioritization.
