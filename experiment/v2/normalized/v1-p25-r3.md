## angles
- Information about engineering changes arrives late and at angles to documentation, missing pieces -> this distortion happens at the unmanaged boundary between engineering (fast-moving) and documentation (slower) -> design a defined crossing point in the release process, such as a "doc impact" field in the PR template, to give information a managed interface rather than letting it refract uncontrollably
- A single release carries multiple types of change (UI copy, auth behavior, API contract, deprecation) that belong in different destinations (help center, README, Confluence) -> without designed routing, they scatter across doc locations, some landing correctly and others becoming outdated in wrong places -> define the routing geometry once (e.g., a field in the PR template mapping each change type to its destination) so that the sorting happens at the interface, not after the fact
- Changes that are deep, implicit, and not explicitly surfaced in their description never reach the documentation surface at all, silently breaking docs -> these high-angle approaches to the doc interface are analogous to total internal reflection, where energy doesn't cross the boundary -> intervene either at the source (required "does this change user-facing behavior?" checkbox in PR templates) or at the interface (file-path triggers in CI that flag changes to sensitive areas like /auth/ or /api/ as doc-review required)
