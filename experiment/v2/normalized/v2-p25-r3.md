## angles
- Documentation surfaces respond at different rates to the same engineering change signal → different surfaces (help center, README, Confluence) serve different audiences and move at different speeds → assign explicit SLAs by audience type: customer-affecting changes get help center updates within 48 hours of shipping; developer-facing changes get README updates in the release PR; internal changes get batched into monthly or quarterly Confluence sweeps, preventing customer-affecting changes from aging in a general backlog.
- The boundary between engineering and documentation is positioned after launch, causing information to reflect back into engineering → most documentation work is triggered only after support tickets arrive → embed a doc-impact checklist (3–5 yes/no questions like "does this change an API endpoint?" or "does this affect customer-facing setup?") directly into the PR template as a machine-readable signal that surfaces documentation needs before release.
- The help center search algorithm ranks archived pages above current ones → this is a property of the search medium itself, not the content → add a "last-verified" date field (distinct from "last edited") to every article and configure the search tool to demote articles not verified within 90 days, causing archived pages to stop ranking above current ones without requiring re-authoring.
