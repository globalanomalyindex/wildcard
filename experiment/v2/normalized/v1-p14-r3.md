## angles
- The 140 WCAG failures are a summed record governed by different constraints (legal review, CSS-only, shared infrastructure) rather than a single body of work -> velocity classes are as distinct as independent harmonic constituents; some are batched and slow (legal gate, ~6 weeks), others are parallelizable and fast -> partition failures by governing constraint first so fast-path work runs concurrently while legal review proceeds, avoiding slow-path tempo setting deadline risk.
- The shared header code nobody owns is the reference substrate for every screen; unfixed focus-management or landmark failures in it propagate across all 60+ screens -> fixing individual screen failures while the datum drifts produces higher fix counts but contaminated user experience for screen reader users on the header, which ships with every deploy -> establish shared-header governance (one named owner, one review step per sprint) before fast-path fixes start shipping to clarify what "done" means for a large fraction of the 140.
- The legal review cycle is a planned gap structure where batches suspend for six weeks -> fixes with cross-dependencies on wording changes (legal gated) coupled to unlicensed fixes (shippable now) risk partial remediation and integration rework when batches clear -> identify which non-legal-gated fixes are tightly coupled to in-flight legal-gated changes before submitting batches, as a pre-flight check before the gap.
