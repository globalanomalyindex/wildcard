## angles
- Voicers triage by finding which failures interrupt flow in the compass range, not by fixing each one in isolation -> the 140 WCAG failures should be mapped not by severity level but by task-flow position (entry form / mid-application / submission / error-recovery) to identify which are breaking the actual application flow for users in constrained conditions -> prioritize failures at entry screens, form-submission paths, and error states in the first legal batch and first sprint regardless of WCAG level
- Nicking changes onset, not pitch; structural markup changes have no text impact and require no legal approval, unlike wording changes -> audit the 140 failures for a binary: does remediating it require changing text a user reads, any disclosure, or legal notice -> split into a legal batch (30–50 failures requiring approval) and a technical track (60–80% of failures that are structural/markup only) that can run in parallel without adding approval overhead
- The room is part of the organ; pipes voiced in the workshop fail when deployed in the actual acoustic -> fixes verified in development (desktop Chrome, recent NVDA/VoiceOver, broadband) will break in actual deployment (five-year-old Android + TalkBack, library computers + JAWS, throttled data) -> add three smoke-test checkboxes to definition of done: TalkBack on Android API 28+, 400kbps bandwidth throttle through form submission, and 15-minute session completion window
