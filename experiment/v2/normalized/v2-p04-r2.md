## angles
- Messages past relevance threshold are being deserialized and scored unnecessarily -> The problem is full CPU spent on records already useless by arrival time -> Before calling JSON deserializer, check record timestamp header against current time; if older than 60 seconds, commit offset and skip (no deserialization, no scoring, no write); turns 50-minute backlog drain into fast-forward.
- Lag is being treated as a status metric rather than a change signal -> The problem is reacting once damage is done (messages aging past deadline) -> Monitor delta between broker ingestion rate and consumer pull rate in real time; trigger mitigation when delta turns positive and holds for 30 seconds, providing lead time of tens of seconds instead of tens of minutes.
- Lag is presented as a single aggregate cluster metric -> The problem is assuming uniform load when some partitions likely carry disproportionate load (uneven key distribution) -> Log per-partition consumer lag across all 48 partitions during peak hours for one week; if top 20% of partitions account for more than 60% of lag, the issue is structural hot-key distribution, requiring isolation to dedicated topic/partition set with tuned consumer group, not horizontal scaling.
