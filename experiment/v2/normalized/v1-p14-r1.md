## angles
- The 140 failures are not uniformly distributed; some completely block tasks (missing form label = screen-reader user sees nothing) while others degrade experience but remain navigable. Sorting failures by "does this stop the task completely for AT users" before sorting by WCAG criterion reveals a threshold between blocked and degraded, and clusters at that threshold deserve different treatment than the long tail. -> Maps onto treating all violations uniformly when some are blocking and others are degrading. -> Score and sort failures by blocking severity on task paths before sorting by compliance category.
- A WCAG audit produces violation counts, but the count doesn't reveal the gradient of where user experience is most blocked. The gradient you care about — where specific user populations fail to complete tasks — is invisible in a violation list and must be inferred from proxy signals: which failures appear together on high-traffic screens, which criteria are overrepresented on specific application paths, which failure types are known to be session-ending. -> Maps onto relying on raw violation counts as a guide to priority without measuring actual user-path blocking. -> Reposition measurement probes: score by task-path density, user-constraint severity, and known session-ending failure modes instead of violation frequency.
- Your three-vendor structure with shared unowned header code is stratified — each vendor develops their own WCAG handling equilibrium (interpretations, fix schedules, component patterns), and the unowned shared layer blocks mixing between them. As layers diverge, eventual alignment becomes harder. -> Maps onto stratification in accessibility handling that increases with time and divergence. -> Identify one high-impact shared component (skip-nav, focus trap, error announcement region) that all three vendors implement consistently before layers diverge further — this stops the divergence clock and maintains a mixing boundary.
- Vendor boundary failures and shared-but-unowned code failures are structurally invisible to any single vendor doing their own accounting; the seams are predictably unclaimed. Additionally, different failure types (technical vs. content) maintain the non-compliance gradient at different rates — legal batch cycles keep content failures in place at one cadence, technical failures cycle differently. -> Maps onto treating technical and content failures on the same remediation timeline when they respond to completely different interventions. -> Track technical and content failures on separate flow rates and separate remediation streams, because they have different response cycles and mixing them may make deadlines feel uniformly hard when parts of it are not.
