Drawing on weather-station siting work brings three moves to mind — take or leave any of them.

**A Stevenson screen covers only part of the instrument, so the recorded temperature drifts from the true ambient.** In siting practice, the response is not "extend the screen to cover everything" — it is "identify which stations are producing records that actually get used, and bring those up to standard first." The 156 components missing from your React package are not equally costly: some are used in every sprint, some were built for one edge case and never touched again. A usage-weighted triage — pull the last three months of detach events from Figma's version history, list which components designers detached and rebuilt by hand, and rank them by frequency — gives you an ordered repair list. Fix the top 10 or 15 components that caused the most manual workarounds. That is the work that eliminates the most production drift in the shortest time, with no headcount. The other 141 can wait.

**A correctly built screen that is badly sited still yields biased readings — and the siting error is invisible to the instrument.** Your customer flagged focus states specifically. That is not a coverage-gap problem; it is a siting problem inside components that already exist in the React package. A targeted audit of every interactive component — buttons, inputs, links, selects, modals — checking whether each has a focus-state prop, whether that prop maps to a design token, and whether that token matches the Figma library value — would produce a defect list you can work from directly. If you run that audit now, you likely have a fixable list within a week, and you can close the customer's specific flag before it surfaces in their next review.

**The record looks continuous but is broken.** In station networks, a microclimate change near a sensor — a new road, a new building — breaks the homogeneity of the time series without any visible discontinuity in the data. The Figma library and React package still look like one system from the outside, but every sprint where designers detach and engineers improvise, the gap widens invisibly. The fix in the station world is a metadata log: record the date and nature of any site change so the break is visible and can be corrected for. A lightweight equivalent in your workflow would be a running detach log — at the end of each sprint, a designer checks Figma version history for detach events and logs them (component name, reason, what prop or variant was missing) in a shared doc or Notion table. This is about thirty minutes of work per sprint. Over two months it gives you real evidence of where the React package is failing designers, a prioritized backlog with actual frequency data, and a visible metric you can show leadership that connects "detaches per sprint" to "production inconsistency" — a proxy for consistency that can sit alongside velocity without requiring headcount to track.

These are optional moves; you know your constraints better than I do. Happy to pull any of these threads further, or step back out.
