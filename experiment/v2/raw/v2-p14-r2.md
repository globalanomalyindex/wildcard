Drawing on the work of pipe-organ voicing — specifically nicking flue mouths so that every pipe speaks evenly across the compass — three things surfaced that might be useful here.

---

**The compass-range triage.** A voicer's first pass over a rank isn't about perfecting each pipe; it's about finding the ones whose onset is wrong in a way that interrupts the flow of the scale. You run the notes and stop the moment something sounds uneven. The goal is not "fix everything" — it's "find what's breaking the speech pattern."

Your 140 WCAG failures are naturally framed as a list to work through. But the structural question is: which failures create an interruption in the actual task flow — applying for housing or food assistance — that users in constrained conditions will hit, and which are intonation problems deep in the compass that almost no one reaches? Map the 140 failures not by WCAG level (A/AA/AAA) but by task-flow position. Build a spreadsheet with: failure ID, screen, task-flow stage (entry form / mid-application / submission / error-recovery), and user-context flag (screen-reader critical / bandwidth-sensitive / session-timer critical). Failures at the entry screens, on form-submission paths, and inside error states are the pipes that play on every scale — they need to be in the first legal batch and in the first technical sprint regardless of WCAG severity level. This prioritization costs one sprint and doesn't require any approvals; it repoints the nine months of work toward the failures users in your actual population will actually encounter.

---

**Nicking controls onset, not pitch.** The nicking distinction is this: nicking doesn't change the note the pipe plays — it changes how the pipe gets there. The note is pitch; the onset is onset. In the portal, the analogous distinction is: wording changes change what users read — those require state legal approval, roughly six weeks per batch. But most WCAG failures are not wording failures. ARIA roles and attributes, focus order, color contrast ratios, missing `aria-label` on form inputs (where the label text is already visible and legally approved — only the markup association is missing), keyboard traps, skip-nav links — these are structural markup changes with no legal language. The displayed text does not change; the approval requirement doesn't apply.

Audit the 140 failures for a simple binary: does remediating this failure require changing any text a user reads, any disclosure, or any legal notice? If yes → legal batch. If no → technical track, no batch required. In typical WCAG audits, roughly 60–80% of failures fall into the structural/markup category. If that holds here, you have two parallel tracks: one legal batch containing maybe 30–50 failures, and a technical track you can start immediately. Running them in parallel doubles effective throughput within the same nine-month window without changing the approval cadence or adding vendor coordination overhead.

---

**The room is part of the organ.** A failure mode voicers fear: pipes voiced in the workshop sound right in isolation but fail in the deployment acoustic. The room is not separate from the instrument — the reverberant space loads each pipe and changes how it speaks. Voicing that ignores the deployment environment fails on the day it matters.

Your fixes will be verified in development environments — likely desktop Chrome, likely with NVDA or VoiceOver on a recent OS, likely on a broadband connection. Your deployment acoustic is a five-year-old Android phone with TalkBack, a library computer with JAWS, and a prepaid data plan throttled at month-end. Some ARIA patterns that work in Chrome + NVDA break in TalkBack on Android API level 28 (Android 9 — still the modal device in low-income populations in most states). Some multi-part form pages that render fine on broadband leave a screen-reader user mid-form with no recovery path at 400kbps.

Add three checkboxes to the definition of done for every fix before it closes: (1) smoke test with TalkBack on a real or emulated device at Android API level 28 or older; (2) smoke test under a 400kbps bandwidth throttle in browser DevTools, through a complete form submission; (3) smoke test that a form application can be started, completed, and submitted within a 15-minute window. These are not a separate QA sprint — one person, one hour per ticket cluster. The cost is small; the benefit is that your court documentation reflects fixes verified in conditions that resemble your actual users' conditions, which is considerably harder to contest than fixes verified on developer hardware.
