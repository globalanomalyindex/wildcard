I'm a physicist who works on energy propagation across interfaces - specifically, the optics of what happens when a wave crosses from one medium into another with a different propagation speed. My lens today is energy-and-flow: not the wave itself, but how the crossing changes it.

The phenomenon I spend my days on is refraction. What's strange about refraction is that the distortion doesn't happen inside the medium - it happens at the boundary. Both media can be perfectly well-behaved; the bending is entirely an interface phenomenon. Manage the interface, and you manage the path of the energy. Leave it unmanaged, and the energy still crosses - but it crosses at an angle you didn't choose, in a direction you didn't intend.

Looking at your situation through that lens, a few things genuinely rhyme.

---

**The information is refracting at an unmanaged interface**

In my field: when a wave crosses from a fast medium into a slow one, the path bends sharply. The bending is entirely determined by the speed differential - and it's predictable, once you know the geometry of the interface.

What maps here: engineering moves fast; documentation is a slow medium. Information about a change - an auth flow rewrite, a new token scope - crosses from the engineering side into your world, and it bends. It arrives late, at an angle, missing pieces. But the bending isn't random - it's determined by the interface geometry, which right now is undefined. There is no designed crossing point. The release process has no natural boundary where "information that needs documentation review" gets shaped for transit.

The thing that transfers: in optics, you don't fix refraction by asking the light to be more careful. You redesign the interface. A single, defined crossing geometry - say, a "doc impact" field in the PR template that engineers fill in before merge - is the interface becoming designed rather than accidental. You're not changing the source or the destination; you're giving the energy a managed crossing point.

---

**Dispersion: the same release refracts differently depending on what it carries**

In my field: white light looks unified at the source, but at an interface, each frequency bends at a slightly different angle - because each frequency travels at a slightly different speed in the second medium. Without designed geometry, they scatter. A prism makes the same dispersion useful, because its geometry is calculated: each frequency's path is known and each lands somewhere specific.

What maps here: a single release carries multiple types of change - UI copy, auth behavior, API contract, deprecation. These don't all belong in the same doc location; they're different "frequencies" headed for different destinations (help center, README, Confluence). When they arrive at an unmanaged interface, they scatter: some land in the right place, some land nowhere, some land outdated in the wrong one.

The thing that transfers: the intervention isn't "document everything in one place and sort it out later." It's closer to the prism move - define the geometry such that each type of change has a known path to a known destination, before any individual release arrives. A routing key in the PR template (auth flow → help center; API contract → README; internal process → Confluence) means the "frequencies" are already separated at the interface. What you're currently spending to sort after the fact might be reducible to defining the geometry once.

---

**Total internal reflection: steep angles of incidence never cross at all**

In my field: when a wave hits an interface at too steep an angle - more parallel to the boundary than perpendicular to it - it doesn't cross at all. It bounces back entirely. This is total internal reflection, and it kicks in predictably at a critical angle. The energy is trapped in the original medium.

What maps here: not all changes arrive at the doc interface at the same angle. Some changes are surfaced and explicit - they hit the interface nearly head-on, and some version of them reaches you. But other changes are steep-angle: a backend PR that rewrites the session token validation logic, described entirely in code and reviewed entirely by engineers, makes a near-zero-degree approach to the documentation surface. It never crosses. You don't hear about it. The auth guide silently breaks.

The angle, in this framing, is how explicit and surface-facing the change description is. The deeper and more implicit the change, the more it reflects back into the engineering medium without crossing.

The thing that transfers: in optics, you prevent total internal reflection either by changing the angle of incidence at the source, or by using an anti-reflection coating at the interface. The source-side move would be something like: a required "does this change user-facing behavior?" checkbox in the PR, which forces even the steep-angle changes to present themselves head-on. The interface-side move is a lightweight mechanism that catches anything touching authentication, billing, or other high-sensitivity areas regardless of how it was described - a file-path trigger in CI that flags any PR touching `/auth/` or `/api/` as "doc-review required." You might find it useful to think about which of your silently-broken docs were steep-angle arrivals, and whether the intervention is at the source or at the interface.
