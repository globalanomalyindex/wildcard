## angles
- Staffing for average load during burst arrival windows is a structural mismatch; airport ground operations pre-stage burst capacity (cleared holding areas, triage separation) rather than hiring for averages -> Saturday morning intake is structurally identical to airport flight bursts, where two mechanics handling average daily throughput cannot absorb concentrated equipment returns -> design Saturday intake around the burst pattern, separate obvious fast-path items (5-minute sign-offs) from slow-path ones (fuel low, damage disputes) to prevent everything from forcing into the same queue
- FIFO service at a bottleneck under burst load systematically starves items that take longer to process, keeping them deferred while faster-cycling items pass through -> excavators dwell unreturned because they are the slow-service item class, deferred during intake triage and pushed behind faster-cycling skid steers and lifts; aggregate utilization looks fine because fast-cycling assets carry the numbers -> institute a hard scheduling constraint (no asset class waits more than N days without an inspection pass and status review) mirroring airport priority routing
- Treating return receipt as proof of inspected availability conflates two separate facts (presence vs. inspected state) -> units reach available state in the system on return timestamp rather than on inspection close-out, making promises against phantom certainty, like carousel updates on manifest receipt before physical scan -> gate availability on an inspection record close-out, not just return timestamp; require a system-level "inspection pass" tap before status flips, decoupling manifest from physical state as two separate data events
- Process quality degrades under burst load unless the process is pre-staged as a constraint, not invoked from memory -> check-in photo inconsistency occurs because the protocol is invoked from memory under return pileup pressure, and discretionary steps get compressed or skipped, like airport baggage handlers under burst load -> pre-stage the intake checklist as a mandatory form opened on the device before customer arrival, with fixed prompts (fuel gauge photo, left side photo, damage declaration) that must be completed before return is logged, so burst pressure doesn't degrade documentation quality
