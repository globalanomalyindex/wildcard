## angles
- Lack of credential inventory before management strategy -> Before choosing a vault or policy, conduct a pure discovery pass (scan git history, CI variables, config vars) to map what credentials exist, which services they unlock, and rotation status -> Audit-only inventory script creates an artifact SOC 2 auditors will require anyway, addressing the root cause (missing map) rather than symptom (blind rotation)
- All credentials treated as equally catastrophic -> Scope credentials by blast radius (isolate subsystems so compromise is limited) and use short-lived credentials (AWS IAM roles via OIDC, Heroku buildpack-level injection) that expire before attackers can pivot -> Reframe short-lived tokens as "session-only" rather than tooling burden; propose OIDC-based AWS credentials for CI first to remove static secrets invisibly
- Setup friction in local dev environments -> Adopt project-level `.env.example` with fake-but-valid format values plus a one-command bootstrapper (Makefile target or shell script) that pulls real dev credentials from a single source (dev vault path or AWS SSM Parameter Store) -> Engineers run one command once, get working local setup, production credentials never touched locally, single audit trail for credential pulls
- Unexpected loss of keyholders and unknown credential scope -> Create lightweight succession document per service (credentials needed, location, owner, rotation procedures) as both incident runbook and SOC 2 evidence -> Turns contractor-laptop incidents from two-day outages into thirty-minute exercises without requiring immediate tooling changes
