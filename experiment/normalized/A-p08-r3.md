## angles

- Peak-count alerts measure exceedance frequency rather than accumulated harm, but action rate of 10% indicates alerts are decoupled from actual payment-path risk -> Instead of asking "did the value cross the threshold?", ask "does this host's pattern over the last N minutes look like the patterns that historically precede payment-flow events?" -> Reframe alert logic to measure dose (accumulated risk pattern) rather than peak exceedance

- Payment-flow failures require one criterion (non-negotiable response), while other system failures require a different criterion -> Current infrastructure has inherited the strict criterion across all alert types by default -> Explicitly separate criterion A (payment-flow) from criterion B (other alerts), giving each its own policy, threshold, and response guarantees

- No intervention strategy should be attempted without an exposure map showing which alerts actually correlate with payment degradation -> Join historical alert telemetry against payment-path outcome data, not just postmortems, to discover which alert types correspond to real degradation and which are noise -> Empirically identify which alerts genuinely protect regulated paths before adjusting thresholds

- Engineers muting alerts overnight is a symptom of failed upstream controls, not a problem to address at the individual level -> The fact that this rational self-protection became necessary signals how far unresolved the architectural problem has become -> Treat escalating PPE adoption (muting behavior) as a lagging indicator requiring investigation of root causes in engineering controls, not as a manageable state
