## angles
- Heavy exports leave a shared resource state (cache, memory, I/O, CPU scheduling) that persists after completion, degrading arrival conditions for subsequent tenants -> The recovery window is a measurable physical number on a quiescent cluster that serves as a floor for start-interval decisions, revealing the true blast radius and whether staggered tenants actually clear each other -> Measure the recovery curve before building scheduling policy, not after; the policy is only as good as the number underneath it.
- Enterprise tenants cluster exports at the top of the hour (cron defaults) with no reason to do otherwise, creating stacked demand on the channel -> The structural move is export schedule negotiation rather than rate limiting or cluster isolation; offer tenants an offset like :15 or :45 and guarantee a quieter cluster state -> Ask whether spikes are a capacity problem or a scheduling problem that looks like capacity because demand shape was never examined; course architecture, not enforcement.
- Attribution gaps (missing tenant IDs in parts of the request path) are invisible under code-level analysis but surface under concurrent multi-tenant load; context-propagation paths that work fine at low concurrency may fail under thread-pool saturation or async boundary crossings -> The eddy trap analogy: run the attribution audit under realistic load by replaying production traffic with instrumentation rather than static code reading -> Inconsistency may emerge only under specific concurrency conditions that dry analysis wouldn't surface.
