## angles
- Retrieval fails because source-channel metadata (Instagram save / browser bookmark / screenshot / card / notes app) is reconstructed from memory each time instead of recorded as a durable intake field -> without recorded provenance chain, the corpus is unaddressable -> make source-channel a mandatory intake field at capture time, storing retrieval paths in the corpus rather than in memory
- Grandmother's handwritten cards and Instagram saves are high-risk formats: physical paper exposed to loss and degradation, platform-hosted files subject to link rot and deletion -> these have different urgency than the rest of the collection -> prioritize targeted triage of high-risk originals (photograph/transcribe cards, save Instagram recipes as text) before organizing anything else
- Post-cook outcomes like "everyone liked it" are episodic memories that erode as the collection grows, not recorded facts -> untethered assertions fail without a reproducible citation path -> treat outcome as a datable field in the corpus (e.g., "approved, all five, gluten-free variation worked, 2025-11-17") rather than an impression in memory
- Collection lives across five silos (URLs, images, physical cards, app entries, platform saves) with separate retrieval methods, making a week's plan unable to query spices across them -> there is no single addressable corpus -> create a normalized working set alongside originals, linked back to originals via intake-form metadata, so the collection becomes one queryable thing
