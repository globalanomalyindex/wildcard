## angles
- The signal (410 episodes of audio) is intact but completely inaccessible without a transformation step → the audio-to-text transformation is not a nice-to-have, it gates everything downstream (publisher timestamps, guest lookups, future use) → prioritize starting the transformation pipeline immediately, because everything depends on it and it cannot be paralleled around.
- The naming inconsistency (some files by date, some by guest) prevents knowing which episodes the publisher needs → before running transcription jobs, do a lightweight metadata pass to build a flat index (filename, folder, duration, detectable date/guest) that enables sequencing → this inventory is cheap to produce and lets you queue the publisher's specific episodes without waste.
- A transcript without word-level or segment-level timestamps satisfies the "we have text" check but fails the accountability check → the publisher is asking for a chain-of-custody record (this quote came from this file at this offset) → choose a transcription tool whose output format includes word-level timestamps embedded in the file, because retrofitting them from plain text is significantly harder than getting them right the first time.
