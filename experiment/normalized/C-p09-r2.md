## angles
- Enterprise tenants already have contracts specifying latency SLAs but not scheduling rights -> the lever could be commercial rather than infrastructure -> offer fee credit or expanded API quota in off-peak windows in exchange for tenants agreeing to stagger exports to off-peak times
- Tenant IDs are inconsistently threaded so rate limiting cannot be done today -> preemption (priority-based dequeuing) inverts the problem of rate limiting -> mark scheduled-export queries with a low-priority tag and let ClickHouse's workload management bump them behind interactive queries
- SLA contracts promise a latency number but say nothing about when load is measured -> reframe SLA as rolling average over 15-minute window rather than instantaneous p99 to smooth the spike mathematically -> audit contract language with legal team, as "p99 latency" measurement windows are rarely defined to the millisecond
- Exports are scheduled jobs generated by code, so there is a natural trigger point (export API endpoint, UI button, or cron runner) -> the right place to absorb variance is upstream at the bottleneck trigger, not downstream at the query layer -> add token-bucket at export-initiation endpoint with concurrency limit per tenant per hour
