## angles
- Structured noise sources in degraded recordings are periodic, predictable, characterizable signals with their own time-frequency signature, modeled out explicitly before searching for signal -> CPU and disk threshold alerts during batch jobs are known, periodic, characterizable event classes with a time signature, occupying the same channel as events that matter -> Build a forward model of the batch-correlated alert pattern (hosts, metrics, time windows, amplitude envelope) so that alerts outside that model stand out structurally, separating the noise floor from the signal band without changing thresholds.
- Detection cannot raise thresholds globally, so separate impulsive events into separate detection channels by source type, each with its own criterion calibrated to that source's characteristics -> Compliance-critical payment-flow alerts and batch-job-correlated infrastructure alerts are not the same source class and should not share a threshold policy -> Classify each current alert by source type — is this alert capable of being triggered by a batch job, or can it only fire when a payment flow anomaly occurs — then tune aggressively on the first class without touching the second.
- Provenance tracing works backward to build the causal map of what could have fired and what couldn't have before making any detection decision -> The problem of not being able to confidently say which alerts protect regulated paths is the absence of a causal map; the team is analyzing waveforms without a maintenance log -> Do a one-time provenance audit before any tuning decision: for each alert type, trace it backward to its triggering mechanism and ask whether that mechanism can be activated by a routine batch job, yielding the causal trace from alert to regulated path that compliance needs.
