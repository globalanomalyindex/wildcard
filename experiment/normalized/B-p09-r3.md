## angles
- Destructive load in cyclone systems concentrates in the narrow eyewall (10-20% of radius) not the wide outer bands -> Population of 4,000 tenants has same structure, with small fraction generating top-of-hour batch exports as the load concentration zone -> Don't instrument every tenant; instead target the structural feature (scheduled/batch patterns) with partial instrumentation like pattern-based query detectors (duration threshold, burst-within-window) to get attribution at impact point before full end-to-end tenant ID propagation
- Cyclone intensity decisions from peripheral data (satellite temps, rainband structure) are systematically misleading; only direct inner-core measurement (reconnaissance aircraft, dropsondes) gives reliable signal -> Current observability is peripheral proxy (p99 latencies, cluster-level aggregate metrics) that signal intensification but not where the eyewall is; per-query attribution at cluster entry point is the dropsonde -> Need direct inner-core measurement of who asked, when, what shape—without requiring full tenant-ID propagation everywhere before acting
- Cyclone rapid intensification occurs when system deepens faster than observation network resampling, making reported intensity structurally stale from decisions made on readings several hours old -> System has same lag structure architecturally: no real-time signal of responsible tenant during spike, post-mortem runs after SLA window closes -> Reconnaissance flights happen during quiet periods because window to act during intensification is too short; build tenant attribution instrumentation during quiet hours so next top-of-hour event is already tracked in real time as it begins
