# wildcard landing page — design spec

**Date:** 2026-06-10
**Status:** Approved (design); pre-implementation
**Scope:** Phase 2 — the landing page in `site/`. (The skill itself shipped in phase 1.)

---

## 1. The big idea — the page is itself a draw

The page that explains seeded non-linear structure is itself a seeded non-linear structure.
On every visit, a **seeded guillotine engine** subdivides the below-fold space using only the
three ratios from the user's chickpea grid legend — **1/φ (0.618), 1−1/φ (0.382), 1:1** — and
deals the content sections into the resulting cells. Different on every visit; reproducible via
`?seed=<anything>`; rerollable in place. Readability is guaranteed by **invariants the engine
cannot violate** (§4), not by hope. The hero viewport stays authored to the Figma; the engine
owns everything below it.

Everything "live" is real: real entropy, the real 344-domain map, real session transcripts.
Nothing is simulated for effect (medium matches message — only if real).

## 4. The cell engine

### 4.1 Module seam (pure, testable)

`site/js/phi-grid.js` — **pure data in → pure data out, zero DOM**:

```
plan(seed: uint32, viewportClass: 'desktop'|'narrow', manifest: SectionSpec[]) → CellTree
```

`SectionSpec` = `{ id, minColFrac, minRowPx, kind: 'text'|'image'|'compact' }`.
`CellTree` = nested guillotine splits `{ dir: 'h'|'v', ratio: 0.618|0.382|0.5, children }` with
leaf cells `{ section?: id, empty?: true }`. A thin `site/js/render.js` maps the tree to CSS
Grid; all visual styling lives in CSS.

### 4.2 Invariants (the engine cannot emit a layout that violates these)

1. **Order**: traversing leaves in reading order (top-to-bottom, then left-to-right by cell
   origin) yields the manifest order exactly.
2. **Fit**: every text cell ≥ its `minColFrac` of the content column and ≥ `minRowPx`; body
   text measure stays within 38–70ch (cell width + typography together guarantee it).
3. **Closure**: guillotine cuts only → no gaps, no overlaps, by construction.
4. **Ratios**: every split ∈ {0.618, 0.382, 0.5}. Nothing else.
5. **Bounded emptiness**: 2–5 empty cells; no two adjacent empty cells wider than the φ-column.

Generation is rejection-sampled: draw a candidate tree, validate, re-roll internally (≤ 32
attempts), then fall back to a **committed known-good layout** (the output of a blessed seed,
baked as JSON) — the page can never render broken.

### 4.3 Seeds and streams

- Per-visit layout seed: `crypto.getRandomValues` u32. `?seed=<string>` overrides (hashed via
  the same CRC as §5.3); reroll button redraws and writes `?seed=` via `history.replaceState`
  so any layout is shareable.
- Stream separation mirrors `draw.sh`: layout indices come from `crc("layout:"+seed+":"+n)`,
  demo draws from `crc("domain:"+seed)` / `crc("lens:"+seed)` — independent streams, one seed.

### 4.4 Narrow viewports and no-JS

- `< 760px`: engine inactive; authored single-column flow (document order), hairline rhythm
  retained via CSS. No dice on mobile — readability over theater.
- No-JS at any width: the same single-column flow (the engine is enhancement, not dependency).
  All content present and styled; demo column shows the three recordings (static) and a note
  that the die needs JavaScript.

## 5. The demo column (authored, inside the hero)

### 5.1 Live draw

Die button → `crypto.getRandomValues` u32 → uniform index over the committed domain list →
typed-out reveal of `domain=…` + `lens=…` chip. A persona sketch cannot be generated without an
LLM, so the reveal shows **exactly what the mechanism produces**: the domain, the lens, and a
one-line invitation ("now imagine what a `<domain>` specialist would notice about your project —
that's the part that happens in Claude Code"). Honest caption: the page rolls the dice; the
noticing happens in Claude Code.

### 5.2 Data provenance (no hand-typed data)

`scripts/gen_site_data.sh` regenerates `site/js/domains.js` from `references/domains.txt`:
preserves **line order** (index parity with `draw.sh`), strips axis tags, exports
`DOMAINS` (344 strings), `LENSES` (parsed from `scripts/draw.sh`'s LENSES line — single source
of truth), and a provenance stamp (source file, count, generating commit). CI-less guard: a
node test fails if `domains.js` count ≠ `domains.txt` count.

### 5.3 Seed parity (the falsifiable flourish)

`site/js/entropy.js` implements **CRC-32/CKSUM** (POSIX cksum: poly 0x04C11DB7, init 0,
unreflected, length-appended, final XOR) so that for any seed,
`crc("domain:"+seed) % 344` picks the **same expert in the browser as
`bash scripts/draw.sh --seed <seed>` in a terminal**. A parity test executes `draw.sh` for a
sample of seeds and asserts identical picks. The page says so, and anyone can check.

### 5.4 Field recordings

Three dated, expandable cards beneath the die — abridged from the real validated transcripts
(rush-light dipper ↔ thundering herd; vellum preparer ↔ three-timeline foreshadowing;
pointe-shoe fitter ↔ honest near-abstention on a CSV tool), each labeled "real output from a
real session — 2026-06-10", with the drawn domain, lens, and the strongest seed in the
three-beat shape. Full text in a `<details>` expansion.

## 6. Motion (organic ambient)

- Hero: a canvas layer of drifting pollen/seed particles (periwinkle/gold dots, sparse),
  seeded from the layout seed — same honesty, no `Math.random` theater. Paused entirely under
  `prefers-reduced-motion`; capped particle count; `requestAnimationFrame` throttled when tab
  hidden.
- Cells: fade/translate-in on IntersectionObserver, one-shot, CSS-driven.
- Draw reveal: monospace type-on for `domain=`/`lens=`, ~30ms/char, skippable on click,
  instant under reduced motion.
- No scroll hijacking, no pinning, no parallax.

## 7. Accessibility & quality bar

- Semantic HTML (`header/main/section/footer`), real text everywhere, alt text for the grid
  image, `aria-live="polite"` on the draw result, focus-visible styles, keyboard-operable die
  and expanders.
- Contrast: body `#302e3a` on `#e5e0f6` passes AA. The gold `#d0c382` on seafoam is decorative
  /large-type only (hero 40px row); never used for body text.
- Lighthouse targets: ≥95 accessibility, ≥95 performance (no-build, two fonts, one image).

## 9. Success criteria

- Hero at 1440px is visually faithful to Figma node 1:2 (fonts, sizes, tracking, positions).
- 100 random seeds → 100 valid layouts (all invariants pass in `node --test` property runs);
  visibly different compositions across seeds; `?seed=` reproduces exactly.
- Seed parity: browser pick == `draw.sh` pick for the sampled seed set.
- No-JS renders the full story readably; `prefers-reduced-motion` kills all ambient motion.
- The three recordings read as real and dated; nothing on the page fakes an LLM.

## 10. Out of scope (YAGNI)

- No analytics, no CMS, no backend, no build toolchain, no router.
- No live LLM calls from the page (the honesty stance, not a limitation to apologize for).
- No mobile-specific generative layouts (single column below 760px).
- Deployment automation (GH Pages workflow) — the site is static and relative-pathed; wiring a
  deploy is a later, separate step.
